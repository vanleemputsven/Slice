# Slice — architecture notes

## Inherited from IP-WaterView (conceptually)

- **`src/` layout** with App Router, feature-oriented `components/` folders, and `lib/` for domain logic and validation.
- **Semantic design tokens** mapped into Tailwind (canvas, surface, accent, card shadow)—WaterView uses light teal/aqua; Slice uses a dark neutral + coral accent while keeping the same token *idea*.
- **Zod at boundaries** for subscription records and preferences; no silent parsing of external or database rows without validation.
- **Product shell**: sticky header, `max-w-6xl` content column, composable cards (`.slice-card` / `.wv-card` pattern).

## Modernized choices

- **Next.js 16** + **Tailwind CSS v4** (`@import "tailwindcss"`, `@theme inline`) as generated by `create-next-app`, instead of Tailwind v3 + separate `tailwind.config.ts`.
- **Supabase** (Postgres + Auth) with **Row Level Security (RLS)** so each row is scoped to `auth.uid()`. The browser uses only the **anon / publishable** key; session cookies are refreshed via **middleware** (`@supabase/ssr`). See `supabase/migrations/` for the canonical schema.
- **React Hook Form + `@hookform/resolvers@3` + Zod 3** for add/edit flows (v5 resolvers target Zod 4; this pairing stays stable on Next 16). Native `<dialog>` for a lightweight modal.
- **Recharts** (`recharts`) for dashboard-only client visualizations; aggregates live in `lib/subscriptions/chart-data.ts` so charts stay presentation-only.

## Data model

- Subscriptions are **normalized records** (`SubscriptionRecord`); monthly burn uses **cycle normalization** (monthly / yearly / N-month custom) before share-splitting.
- **Share-Check**: when `shared` is true, `shareCount` is people in the split (including the user); personal monthly share is `(monthly total) / shareCount`.
- **Persistence**: `subscriptions` and `user_preferences` tables in Postgres. A **trigger** on `auth.users` inserts a default `user_preferences` row for new sign-ups. New accounts start with **no subscriptions** (demo data is opt-in via “sample data” in the UI).

## Security notes

- **RLS** is the source of truth for authorization; policies are defined in SQL, not only in the app.
- **Never** embed the Supabase `service_role` key in client or public env vars.
- Middleware uses `auth.getUser()` so the session is **verified** before treating routes as authenticated (aligned with current Supabase SSR guidance).
- If public Supabase env vars are missing, **protected routes** redirect to `/login` with `error=config` instead of failing open.

## Legacy / local demo

- `src/lib/storage/slice-storage.ts` remains for reference; the live app persists to Supabase after sign-in.

## Supabase project checklist

1. Create a project and copy **URL** + **publishable/anon** key into `.env.local` (see `.env.example`).
2. Run the SQL in `supabase/migrations/20260329120000_slice_initial.sql` (SQL Editor or `supabase db push`).
3. **Authentication → URL configuration**: add redirect URLs `http://localhost:3000/auth/callback` and your production `/auth/callback`.
4. If the trigger fails to create with `execute function`, use `execute procedure public.handle_new_user();` for older Postgres builds.

**Note:** Next.js 16 may log that the `middleware` file convention is deprecated in favor of `proxy`. Session refresh still works; migrate when you adopt the new convention.
